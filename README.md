# Clipboard History for macOS

A keyboard-first clipboard history manager for macOS, in the spirit of Windows `Win+V` —
built natively with Swift, SwiftUI and AppKit. Lives in the menu bar, watches the clipboard,
and puts your copy history one keystroke away.

## How is this different from macOS's built-in clipboard history?

macOS 26 Tahoe ships clipboard history inside Spotlight (`Cmd+Space` → `Cmd+4`). This app
exists for what that feature deliberately doesn't do:

| | Spotlight (built-in) | Clipboard History (this app) |
|---|---|---|
| Access | Cmd+Space, then Cmd+4 | One dedicated hotkey (default `⇧⌘C`), instant popup |
| Retention | Max 7 days | Configurable — default 30 days, up to forever |
| Pinned items | No | Yes — survive retention and Clear History |
| Auto-paste | Pastes from Spotlight | Pastes into the app you were working in |
| Item size | ~16K characters | Up to 5 MB per item |
| Storage | Apple-managed | **AES-GCM encrypted on disk, key in your Keychain** |
| Privacy controls | Minimal | Excluded apps, secure-input detection, concealed-type honoring, pause |

## Building and running

Requires macOS 14+ and a Swift 6 toolchain (Command Line Tools are enough; Xcode not required).

```sh
# Build the .app bundle (release) and run it
Scripts/make-app.sh
open build/ClipboardHistory.app

# Or run directly during development
swift run

# Unit tests (wraps `swift test`; adds Testing.framework paths needed on CLT-only setups)
Scripts/run-tests.sh
```

The repo is a plain SwiftPM package — `open Package.swift` in Xcode works too.

**Dependency note:** `KeyboardShortcuts` is pinned at 1.15.0 because newer releases contain
`#Preview` macros that don't compile with Command Line Tools alone. If you build with full
Xcode, you can bump it to 2.x in `Package.swift`.

## Permissions

Neither permission is requested at first launch; both are explained in context.

* **Accessibility** (System Settings → Privacy & Security → Accessibility) — needed only for
  auto-paste (the app synthesizes `⌘V` into the previously active app) and for caret-position
  panel placement. Without it, selecting an item still copies it to the clipboard for manual
  `⌘V`.
* **Paste from Other Apps** (macOS 15.4+/26, currently opt-in preview) — macOS is moving
  toward alerting when apps read the pasteboard programmatically. The app checks
  `NSPasteboard.accessBehavior`, walks you through allowing access if needed, and pauses
  capture with a visible menu-bar state if access is denied. To test the future enforcement:
  `defaults write com.rahimrzayev.ClipboardHistory EnablePasteboardPrivacyDeveloperPreview -bool yes`

## Privacy

* Everything is local. No network requests, no analytics, no telemetry, no accounts —
  and no networking code or dependencies to audit.
* History content is **encrypted at rest** (CryptoKit AES-GCM). The key lives in your login
  Keychain and is never synced to iCloud (`kSecAttrSynchronizable` is never set). Item
  *metadata* (timestamps, source app, pin state) is stored unencrypted; only the copied
  content itself is the protected payload. (Note: the stronger data-protection keychain
  requires entitlements that ad-hoc dev signing doesn't provide — planned for the
  Developer ID build.)
* **Clear History rotates the encryption key**, so cleared data is cryptographically
  unrecoverable from disk or backups.
* Items marked confidential by password managers (`org.nspasteboard.ConcealedType`,
  `TransientType`, `AutoGeneratedType` + known vendor markers) are **never** recorded, and
  capture is skipped while secure input is active (password fields, sudo prompts).
* **Browser-extension gap, and its mitigation:** passwords copied via password-manager
  *browser extensions* (Norton, 1Password, Bitwarden — verified June 2026; extensions cannot
  set pasteboard markers, and the copy is attributed to the browser) would otherwise slip
  into history. A default-on heuristic skips password-shaped text (single mixed-class token;
  exempts URLs, emails, paths, hashes, identifiers) — Settings → Privacy. Fully
  lowercase/letters-only passwords can still slip through; no heuristic can catch those
  without eating ordinary text.
* ~30 known password manager apps (verified bundle IDs incl. legacy/Catalyst variants) are
  excluded from capture by default (editable in Settings), plus a name-based fallback for
  unknown apps whose name contains "password"/"keepass"/"keychain".

## Keyboard reference (popup)

| Key | Action |
|---|---|
| `⇧⌘C` | Open/close the popup (customizable in Settings) |
| type | Search (the search field is always focused) |
| `↑` `↓` | Move selection |
| `↩` | Paste into the previously active app |
| `⌥↩` | Copy only (no auto-paste) |
| `⌘1`–`⌘9` | Paste the Nth visible item |
| `⌘⌫` | Delete selected item |
| `⌘P` | Pin/unpin selected item |
| `esc` | Clear search, then close |

## Current limitations

* Text only — images, files and rich text are v2 (the storage schema is already
  versioned and typed for them).
* Source-app attribution is a best-effort heuristic (macOS provides no public API for the
  pasteboard writer's identity).
* Auto-paste cannot work while Secure Keyboard Entry is active (Terminal option, password
  prompts) — the app falls back to copy-only with an explanatory toast.
* `swift run` (unbundled) can't register launch-at-login; use the built .app bundle.
* The bundle script applies an ad-hoc signature for local use. Distribution requires
  Developer ID signing + notarization (deliberate decision — see spec §2.1).
